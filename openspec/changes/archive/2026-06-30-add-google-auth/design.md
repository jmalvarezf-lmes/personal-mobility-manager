## Context

The app currently has no authentication. All API endpoints are open and vehicles have no owner. The frontend is a single-page React app with no routing — it renders the map directly. This change introduces Google OAuth2 login, a users table, vehicle ownership, and a landing page with navigation.

Key constraints already in place:
- FastAPI backend, PostgreSQL via SQLAlchemy, Alembic migrations
- React + Vite + Tailwind frontend, no router installed
- Vehicles table exists in dev only — no production data to preserve
- Existing security middleware: CORS allowlist, rate limiting, security headers

## Goals / Non-Goals

**Goals:**
- Full backend-controlled OAuth2 flow with Google (no client-side OAuth SDK)
- HttpOnly + SameSite=Strict JWT cookie — maximum CSRF protection without a token
- Auto-provisioning of users on first login via `google_sub` as stable identifier
- Vehicle ownership enforced at the API layer via JWT dependency
- React SPA routing with a protected `/map` route
- Frontend displays authenticated user email in navigation bar

**Non-Goals:**
- Refresh tokens — 24h expiry, simple re-login via Google
- Per-vehicle sharing between users
- Role-based access control
- Vehicle location display on the map (separate proposal)

## Decisions

### 1. Backend-controlled OAuth2 flow (Option B)

The login button is a plain `<a href="/api/auth/google/login">`. The browser is redirected to Google by the backend. The backend exchanges the code, provisions the user, sets the cookie, and redirects the browser to `/`. No Google SDK or OAuth logic on the frontend.

**Alternatives considered:**
- Frontend-first (Option A): frontend receives Google ID token and POSTs to backend. Simpler frontend, but splits auth responsibility and requires the Google Sign-In SDK on the client.

Backend-controlled keeps all auth logic in one place and avoids client-side token handling.

### 2. SameSite=Strict for the session cookie

```
Set-Cookie: session=<JWT>; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=86400
```

With `Strict`, the session cookie is never sent on any cross-site request, defeating CSRF at the browser level. This eliminates the need for a double-submit CSRF token entirely.

**Trade-off:** If a user navigates to the app by clicking a link from another site (Slack, email, browser history), the cookie is not sent on that first request and they appear unauthenticated. They land on `/`, click the map link, and are authenticated. Acceptable for a personal-use app.

**Alternative considered:** `SameSite=Lax` with a double-submit CSRF token cookie. More robust for link-sharing scenarios; adds implementation complexity. Rejected because the personal-use trade-off is acceptable.

### 3. Signed state parameter instead of a state cookie

The OAuth state parameter is used to prevent CSRF during the redirect flow. Storing it in a `SameSite=Strict` cookie would break the flow (the cookie wouldn't be sent when Google redirects back — a cross-site navigation).

Instead, the state is a signed, time-limited token generated by `itsdangerous.URLSafeTimedSerializer`. No cookie needed. The backend verifies the signature and a 5-minute `max_age` on callback.

```python
# At /auth/google/login
state = URLSafeTimedSerializer(JWT_SECRET).dumps({"nonce": secrets.token_hex(16)})

# At /auth/google/callback
URLSafeTimedSerializer(JWT_SECRET).loads(state, max_age=300)  # raises if expired or tampered
```

### 4. authlib as the OAuth2 client

`authlib` provides a clean async-compatible OAuth2 client for Starlette/FastAPI. It handles building the authorization URL, exchanging the code for tokens, and fetching the userinfo endpoint.

**Alternative:** `httpx` + manual code exchange. More control, more boilerplate. Rejected to avoid reinventing well-tested logic.

### 5. PyJWT for session token

The session JWT is signed with `HS256` using a `JWT_SECRET` env var. Payload: `{ "sub": "<user_id>", "email": "<email>", "exp": <now+86400> }`.

Verified on every protected request via a FastAPI `Depends(get_current_user)` dependency that reads the `session` cookie, decodes the JWT, and returns the `User` entity.

### 6. user_id NOT NULL on vehicles

The vehicles table contains only dev data. Adding `user_id NOT NULL REFERENCES users(id)` is safe — the migration can truncate the table or require a fresh registration. No nullable migration path needed.

### 7. FastAPI dependency for auth

```python
async def get_current_user(request: Request) -> User:
    token = request.cookies.get("session")
    if not token:
        raise HTTPException(status_code=401)
    try:
        payload = jwt.decode(token, JWT_SECRET, algorithms=["HS256"])
    except jwt.PyJWTError:
        raise HTTPException(status_code=401)
    user = user_repo.find_by_id(UUID(payload["sub"]))
    if not user:
        raise HTTPException(status_code=401)
    return user
```

Protected endpoints add `current_user: User = Depends(get_current_user)` to their signature. No middleware — dependency injection is more granular and testable.

### 8. Frontend: AuthContext + ProtectedRoute

On app mount, `AuthContext` calls `GET /auth/me`. If 200, stores the user; if 401, user is `null`. `ProtectedRoute` wraps `/map` and redirects to `/` when user is `null`. The nav bar renders conditionally based on auth state.

## Risks / Trade-offs

- **Google Cloud Console setup (out of band)** → The OAuth client ID/secret and redirect URI must be registered before the feature works. Document in `.env.example`. Risk: blocks testing until configured.

- **24h JWT validity window** → If the cookie is somehow extracted (requires device compromise, since HttpOnly prevents JS access), it's valid for up to 24h. Mitigation: HTTPS in production (Secure flag), short enough for a personal app.

- **SameSite=Strict first-navigation UX** → User clicking a shared link lands on the landing page instead of the map. For a personal app this is a minor inconvenience, not a blocker.

- **vehicles truncation on migration** → Existing dev vehicle rows must be deleted before the migration that adds `user_id NOT NULL`. Acceptable for dev; would be a breaking change in production.

## Migration Plan

1. Deploy migration 1: create `users` table (no impact on existing code).
2. Truncate `vehicles`, `vehicle_configs`, `vehicle_locations` (dev data only).
3. Deploy migration 2: add `user_id NOT NULL` to `vehicles`.
4. Deploy backend with auth routes and protected vehicle endpoints.
5. Deploy frontend with routing, landing page, and auth context.
6. Register redirect URI in Google Cloud Console.

Rollback: revert migrations in reverse order; redeploy previous backend and frontend.
